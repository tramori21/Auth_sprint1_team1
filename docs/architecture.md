# Архитектура Auth-сервиса

## Назначение сервиса
Auth-сервис отвечает за:
- регистрацию пользователя (signup);
- аутентификацию (login) и выдачу пары токенов (access + refresh);
- обновление access-токена через refresh (с ротацией refresh);
- выход (logout) с инвалидированием refresh;
- хранение истории входов;
- управление ролями (RBAC): CRUD ролей, назначение/отзыв ролей, проверка прав;
- создание суперпользователя через CLI.

Сервис реализован на FastAPI и работает как отдельный микросервис.

## Компоненты

### API (FastAPI)
- Обрабатывает HTTP-запросы.
- Валидирует входные данные.
- Делегирует бизнес-логику сервисному слою.
- Возвращает ответы и ошибки в согласованном формате.

### База данных (PostgreSQL)
Используется для хранения:
- пользователей;
- ролей и связей пользователь-роль;
- истории входов;
- refresh-токенов (или данных для их инвалидирования, в зависимости от реализации).

Миграции выполняются через Alembic.

### Redis
Используется для:
- хранения данных, необходимых для инвалидирования refresh-токенов и реализации logout;
- кэширования (если применено в проекте по урокам).

### Docker Compose
Поднимает инфраструктуру локально:
- Postgres;
- Redis;
- (опционально) сам API контейнером, если это предусмотрено текущей конфигурацией.

## Потоки данных

### Регистрация (signup)
1. Клиент отправляет логин и пароль.
2. Сервис проверяет уникальность логина.
3. Пароль хэшируется и сохраняется в Postgres.
4. Возвращается созданный пользователь (без чувствительных данных).

### Вход (login)
1. Клиент отправляет логин и пароль.
2. Сервис проверяет учётные данные.
3. Выдаётся access-токен (JWT) и refresh-токен.
4. Факт входа фиксируется в истории входов.

### Обновление токенов (refresh)
1. Клиент отправляет refresh-токен.
2. Сервис проверяет его валидность и статус (не инвалидирован ли он).
3. Выдаётся новый access-токен и новый refresh-токен (ротация).
4. Старый refresh-токен инвалидируется.

### Выход (logout)
1. Клиент инициирует выход (обычно с refresh-токеном).
2. Сервис инвалидирует refresh-токен, чтобы дальнейшее обновление было невозможно.

### RBAC
- Роли управляются через отдельные эндпоинты (CRUD).
- Пользователю назначаются/отзываются роли.
- Проверка прав используется в защитных зависимостях/декораторах API.

## Обработка ошибок (принципы)
Сервис различает:
- ошибки валидации входных данных (400/422);
- ошибки аутентификации (401: неверные данные, истёкший токен, неверная подпись);
- ошибки авторизации (403: недостаточно прав);
- конфликты (409: логин занят, дублирующая роль и т.п.);
- отсутствие сущности (404);
- внутренние ошибки (500), которые не раскрывают внутренние детали.

## Конфигурация
Настройки берутся из переменных окружения (через .env):
- параметры Postgres;
- параметры Redis;
- секрет JWT и настройки токенов;
- порт приложения.

В репозиторий не добавляется файл .env, вместо него используется .env.example.
